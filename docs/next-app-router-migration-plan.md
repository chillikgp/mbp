# My Baby Pictures Static-to-Next.js Migration Plan

## Goal

Convert the current static generated My Baby Pictures website into a standard Next.js 15 App Router project while preserving the rendered website exactly:

- Same URLs
- Same HTML semantics where possible
- Same SEO metadata, canonical URLs, Open Graph tags, Twitter tags, JSON-LD, sitemap, and robots output
- Same images and asset URLs
- Same CSS design system
- Same browser behavior for navigation, lightbox, analytics, inquiry forms, product checkout, uploads, preview, and confirmation

This migration should prepare the codebase for Payload CMS integration without making the CMS responsible for rendering the frontend yet.

## Current Architecture

The current website is a static generated site.

- `package.json` runs `node scripts/build-site.mjs` for builds.
- `npm run dev` serves files with `python3 -m http.server 4173`.
- `scripts/build-site.mjs` reads `content/site.json` and `content/products.json`.
- The generator writes complete HTML files into route folders such as `categories/newborn/index.html`, `portfolio/newborn/index.html`, and `shop/milestone-frame/index.html`.
- Shared CSS lives in `assets/styles.css`.
- Shared browser behavior lives in `assets/site.js`.
- Images live in `images/` and are referenced by absolute paths like `/images/portfolio-newborn-1.jpeg`.
- SEO is generated manually inside the `layout()` function in `scripts/build-site.mjs`.
- `sitemap.xml` and `robots.txt` are generated by the build script.

The site currently generates 51 canonical routes, plus compatibility files for:

- `privacy-policy.html`
- `testimonials.html`

## External Requirements To Respect

Next.js App Router uses the `app` directory for routing and `public` for static assets. Its file conventions include `page`, `layout`, `route`, dynamic segments, optional catch-all segments, metadata files, route groups, `sitemap`, and `robots`.

Payload's official Next integration expects Payload files inside the Next `/app` structure, a Payload config, `@payloadcms/next`, a database adapter such as `@payloadcms/db-postgres`, and wrapping Next config with `withPayload`.

Because Payload requires a real Next app, the static-to-Next migration should happen before Payload is embedded.

References:

- Next.js project structure: https://nextjs.org/docs/app/getting-started/project-structure
- Next.js metadata files: https://nextjs.org/docs/app/api-reference/file-conventions/metadata
- Next.js sitemap: https://nextjs.org/docs/app/api-reference/file-conventions/metadata/sitemap
- Next.js robots: https://nextjs.org/docs/app/api-reference/file-conventions/metadata/robots
- Payload installation in an existing Next app: https://payloadcms.com/docs/getting-started/installation
- Payload Postgres adapter: https://payloadcms.com/docs/database/postgres

## Migration Principles

1. Preserve output first, improve architecture second.
2. Port generator functions into React components without redesigning them.
3. Keep `content/site.json` and `content/products.json` as the source of truth during the first migration.
4. Do not introduce Payload data reads into public pages during this migration.
5. Keep image paths stable at `/images/...`.
6. Keep CSS class names stable so `assets/styles.css` can move into Next with minimal visual risk.
7. Keep existing client behavior intact first, then gradually convert scripts into small client components.
8. Use automated parity checks before removing the static generator.

## Phase 1: Add Next.js Without Changing The Frontend

Tasks:

- Install Next.js 15 with React, React DOM, TypeScript, and required type packages.
- Add `next.config.mjs`.
- Add `tsconfig.json`.
- Add `next-env.d.ts`.
- Update scripts:
  - `dev`: `next dev -p 4173`
  - `build`: keep the static generator temporarily as `build:static`, and add `next build`
  - `start`: `next start -p 4173`
- Move or copy runtime static assets to `public/assets/` and `public/images/`.
- Keep source images in their current form unless the repo decides to treat `public/images/` as the canonical image folder.

Important:

- Do not delete generated HTML yet.
- Do not change CSS.
- Do not change content JSON.
- Do not add Payload yet.

Acceptance gate:

- Next dev server runs.
- `/assets/styles.css`, `/assets/site.js`, and `/images/...` resolve from `public`.
- No public route has been ported yet unless static parity assets are in place.

## Risk Areas

- Font rendering may shift if Google Fonts are changed to `next/font`.
- Moving assets can break `/images/...` or `/assets/...` paths unless placed under `public`.
- Next metadata generation can subtly alter tag order or omit duplicate tags; compare actual `<head>` output.
- React hydration can conflict with the existing global `assets/site.js` if markup changes.
- Product checkout currently stores placeholder order state in `localStorage`; this must remain client-only.
- Portfolio image aspect-ratio classes currently depend on image-size detection in the generator; Next needs an equivalent helper at build time.
- Compatibility `.html` routes need explicit handling.
- Payload should not be introduced until Next route parity is stable.

## Recommended First PR

The first implementation PR should do only this:

- Add Next.js 15 App Router dependencies and config.
- Move static assets into `public` while preserving URL paths.
- Add typed content helpers.
- Add route helper tests or a small parity script that compares `getAllRoutes()` with the current sitemap.
- Port only `/privacy-policy/` and `/testimonials/`.
- Keep the static generator available.
- Do not add Payload yet.

That creates the bridge without putting the whole site at risk.
